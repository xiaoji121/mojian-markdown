# 产品设计规范

> 本规范提炼自墨笺 Markdown 的设计系统实践，方法论源自
> 《Refactoring UI》（Steve Schoger & Adam Wathan）。
> **规则与结构是可复用部分；具体色值/字体是"墨"主题的参考实现**（见附录 A），
> 换一个产品只需替换 token 取值，规则不变。

---

## 0. 五条核心原则

1. **先定性格，再动手**。产品的视觉性格由四个具体因素决定：字体、圆角、颜色、语言风格。四者必须自洽（楷体衬线配小圆角是书卷气，圆体配大圆角是活泼；混搭即性格分裂）。
2. **一切取值系统化**。颜色、字号、行高、间距、圆角、阴影、透明度都从预定义的小集合里选，不在无限池里逐像素纠结。所有取值收敛为 design token，**全项目单一来源文件**。
3. **层级靠颜色和字重，不靠字号**。文字层级用 3–4 级明度灰阶表达；把字号当最后手段，否则主内容过大、次内容过小。
4. **留白优先，边框最少**。分隔区域先用留白，再用背景色差，最后才是边框；一排相同描边的元素就是视觉噪音。
5. **用户可选维度宁少勿多**。每增加一个开关/切换维度都要过一遍消除法：能折进已有维度（例如把"清爽风格"折成一种纸色）就不加新开关；两个互相牵制的选择维度不如一个。

## 1. Token 体系

### 1.1 颜色结构：框架（chrome）与内容（paper）分层

内容型产品（阅读器、编辑器、文档工具）的颜色分两层，**框架和内容纸面解耦**——框架用中性色退到背后，内容像"一张纸浮在工作台上"，而不是糊在一起：

**Chrome 层**（顶栏/工具栏/面板/状态栏），每主题五组：

| 组 | Token | 用途 |
| --- | --- | --- |
| 背景阶 | `--bg` `--bg-2` `--bg-3` `--bg-input` … | 页面 / 面板工具栏 / 侧栏 / 输入框等 |
| 边框阶 | `--border` `--border-soft` `--border-faint` | 强 / 中 / 弱分隔线 |
| 文字阶 | `--text` `--text-2` `--text-3` `--text-4` | 标题强调 / 正文 / 次要标签 / 占位禁用 |
| 强调 | `--accent` `--accent-2` `--accent-ink` `--accent-soft` `--mark-bg` `--sel` | 主强调 / hover 深一档 / 强调底上的文字 / 淡底 / 高亮 / 选区 |
| 语义 | `--good` `--danger` `--info` | 成功 / 错误 / 信息 |

**Paper 层**（阅读/编辑内容面），一套 `--paper-*` token（bg / bg-soft / pre / code / border / text 三级 / accent / mark / sel / 语法高亮四色 / 字体与行距字距），由 `data-paper` 属性切换，可做成**用户可选纸色**（如墨黑/羊皮纸/米黄/纯白/护眼绿）；未选择时跟随框架明暗（暗→深纸、亮→暖纸）。每种纸是**完整配色方案**而非只换底色——文字阶、高亮、代码色都要配套。

规则：

- **一个产品只要一个强调色**，靠 `-2`（深）与 `-soft`（淡底）派生状态，不引入第二品牌色；纸面强调色（`--paper-accent`）是它在各纸色上的明度适配，不是第二个色相。
- 语义色也要进 token；错误红、成功绿散落在代码里就是失控的开始。
- 禁止在业务样式中出现字面色值——出现即说明缺一个 token。

### 1.2 明暗主题（阅读舒适度规则）

- **暗色不用纯黑/近黑**。背景取明度约 10–12% 的深灰（可带品牌色倾向）。近黑底 + 亮字会产生光晕效应（halation），久读眼疲劳。
- **暗色正文降饱和**。用低饱和的暖白灰，不用高饱和的品牌色淡化版——整屏色罩同样致疲劳。强调留给重点句和交互元素。
- **浅色不用纯白**。用纸感底色（奶白至羊皮纸米黄），正文用深棕/深灰而非纯黑；目标对比度 7–10:1 的"清晰但不生硬"区间，不追求 21:1。
- 暗色为默认值写在 `:root`（避免 JS 就绪前闪白）；亮色以 `[data-theme='light']` 覆盖。若产品内有"永远暗色"的页面（如营销首页），亮色覆盖选择器需加作用域限定，防止主题泄漏。
- 无持久化偏好时跟随 `prefers-color-scheme`；用户手动切换后写入 localStorage，多入口（主应用/扩展/弹窗）读同一个键互相跟随。
- 纸色覆盖块（`data-paper`）写在主题块之后，选择器特异度需压过主题块内的纸面默认值（属性双写 `[data-paper='x'][data-paper]` 是同特异度靠顺序取胜的惯用法）。

### 1.3 字体

三个字体栈，各司其职，全产品不出现第四个：

- `--read` 阅读衬线：正文、标题、引用（性格载体）
- `--sans` UI 无衬线：界面控件、面板正文
- `--mono` 等宽：代码、文件名、数字标签、overline

规则：字体栈必须写全回退链（品牌字体 → 系统同类 → generic family）；中文场景补齐系统中文字体。付费/受限字体不入库，构建时按需拉取并子集化。

### 1.4 字号 scale（七档）

UI 界面字号只从 scale 取；**正文内部（h1、code 等）用 em 相对值**，随阅读基准缩放；营销页 display 大字号（>20px 与 clamp()）是唯一例外。

| Token | 值 | 用途 |
| --- | --- | --- |
| `--fs-2xs` | 10px | 徽标、mono 微标签、overline |
| `--fs-xs` | 11px | 状态栏、栏目标题、工具按钮 |
| `--fs-sm` | 12px | UI 默认：动作按钮、列表项、文件名 |
| `--fs-md` | 13px | 面板正文、输入框 |
| `--fs-lg` | 14px | 编辑区、按钮图形字符 |
| `--fs-xl` | 16px | 阅读正文基准 |
| `--fs-2xl` | 18px | 品牌标题 |

不允许出现 12.5px、13.5px 这类"缝隙值"——出现即并入相邻档。

### 1.5 行高（情境化，不与字号绑死）

行高随字号**反比**、随行长**正比**：

- 阅读正文（中文）：2.0–2.1
- 面板/UI 正文：1.6–1.75
- 标题：1.25–1.35
- 单行微标签：1–1.4
- 移动端窄屏正文可降到 1.9

### 1.6 字距

- **中文正文加少量正字距**：+0.02–0.05em（参考实现取 0.04em），显著改善长文呼吸感
- **标题收紧**：-0.01em
- **全大写/字母数字小标签放宽**：+0.1–0.16em
- 等宽数字标签可 +0.03–0.06em

### 1.7 间距

4px 基线：4 / 8 / 12 / 16 / 24 / 32 / 48。设计时先给过量留白再收，不要从挤开始加。间距归属要无歧义——元素与其标题的距离必须明显小于与上一组内容的距离。

### 1.8 圆角（三档）

| Token | 值 | 用途 |
| --- | --- | --- |
| `--radius-control` | 4px | 按钮、输入框、行内码、小卡 |
| `--radius-surface` | 8px | 面板、卡片、弹层、浮动工具条 |
| `--radius-pill` | 999px | 徽标、计数、标签胶囊 |

圆角是性格的载体：全产品统一一套，不允许 3/5/8px 并存。

### 1.9 阴影（三档，双层叠加）

每档由「近距环境光 + 远距投影」两层组成，更有体积感：

- `--shadow-sm` 按钮级；`--shadow-md` 下拉/工具条；`--shadow-lg` 模态/大弹层
- 暗色主题用纯黑基阴影，浅色主题用带品牌暖色的阴影（如 rgba(120,100,60,…)）
- 侧滑面板可用定向阴影（如 `-14px 0 36px`），属排版特例，不入 scale

## 2. 布局与排版

- **行长**：中文正文 30–45 字/行（16px 下内容区约 700–820px）；西文 45–75 字符。沉浸阅读模式也必须收敛到这个宽度，不许铺满屏。
- **长文永不居中对齐**；三行以内的短文案才可居中。
- **不要填满整屏**：导航全宽不代表内容要全宽；边缘留白永远无害。
- **固定宽度侧栏**：侧栏、表单按内容定宽，不随屏幕等比伸缩。
- **窄内容宽屏失衡时拆列**，而不是拉宽。
- **移动优先**：先在 ~400px 画布上成立，再放大。
- **等比缩放是错觉**：跨端时字号、间距、圆角各自独立微调（大屏标题缩小比例应大于正文）。

## 3. 组件规范

### 3.1 按钮金字塔

- **Primary**：实心强调色，一个界面区域只有一个（问自己"用户来这里最主要做什么"）
- **Secondary**：描边，留给次高频动作（通常也只有一个）
- **Tertiary**（默认档）：无边框、次要文字色，hover 时强调色文字 + 淡强调底
- 破坏性动作平时用 secondary/tertiary；只在确认弹窗里才给红色 primary
- 链接不必都上强调色；辅助性链接用字重/下划线弱化表达

### 3.2 其他

- **徽标/计数**一律胶囊（`--radius-pill`）+ `--accent-soft` 底 + 强调色文字
- **卡片分隔**优先背景差与留白；卡片内再嵌边框要克制（避免"框中框中框"）
- **空状态是一等公民**：依赖用户生成内容的界面，空态必须给引导文案 + 行动入口，并隐藏无意义的过滤器/工具
- **用户内容防御**：用户图片放进固定容器居中裁切；浅色用户内容加内阴影/半透明内边框防止与背景融合
- **状态三态**：依赖外部服务的功能（AI/网络）必须有 checking / online / offline 的可视状态与降级文案

## 4. 可访问性基线（不可妥协项）

1. 全局键盘焦点环：`:focus-visible { outline: 2px solid var(--accent); outline-offset: 2px; }`；整面编辑区（textarea/contenteditable）以光标为指示可豁免
2. 全局 `prefers-reduced-motion: reduce` 关停 transition/animation
3. 初始主题跟随 `prefers-color-scheme`
4. 对比度底线：正文 ≥ 7:1（阅读产品自我要求）、次要小字 ≥ 4.5:1、占位 ≥ 2.5:1
5. 不只靠颜色传达状态——配图标或文字（错误=红+文案，成功=绿+对勾）
6. 切换类按钮加 `aria-pressed`，状态区加 `role="status"` / `aria-live="polite"`，图标按钮必有 `aria-label`

## 5. 工程落地规则

1. **token 单一来源**：一个 `tokens.css`，主应用 `<link>` 引入，子端（扩展等）`@import` 构建时内联；禁止任何一端复制色板
2. **禁止内联样式**：模板/JSX 里只写语义类名；样式进 CSS 文件。JS 运行时允许写 `element.style`（开合 display、拖拽宽度、动态字号），类提供初始态、内联覆写动态态
3. **画布类第三方库**（mermaid/图表）不许手抄色板，运行时 `getComputedStyle` 读 token，主题切换时重初始化
4. **`!important` 视为债务**：仅允许出现在"覆盖遗留内联样式"与移动端媒体查询覆写；类驱动后应逐步清除
5. token 更新时同步检查：`<meta theme-color>`、半透明遮罩/渐变里的字面 rgba、manifest 主题色

## 6. 新产品启动检查清单

- [ ] 定性格：字体 / 圆角档 / 强调色 / 语言风格四项书面化
- [ ] 建 `tokens.css`：五组颜色 × 明暗两套、三字体栈、七档字号、三档圆角、三档阴影
- [ ] 暗色底 ≠ 纯黑；浅色底 ≠ 纯白；正文对比度按 4 节核对
- [ ] 中文正文：行高 ≥2.0、字距 +0.02~0.05em、行长 ≤45 字
- [ ] 每个界面找出唯一 primary 按钮，其余降级
- [ ] focus-visible / reduced-motion / color-scheme 三个全局规则就位
- [ ] 空状态、offline 降级、用户内容防御有设计
- [ ] CI 里有构建 + 类型 + 体积约束，样式改动过一遍明暗两主题

### 排版关键值

正文 16px / 行高 2.1 / 字距 +0.04em（经 `--paper-line-height` /
`--paper-letter-spacing` token 化，可随纸面体系整体调整）；
